# 2026-02-05 學習筆記：CMake 流程與 JUCE 核心架構理解

## 1. CMake 建置流程理解 (CMake Build Process)
今天釐清了 CMake 在專案開發中的角色，從以往的手動管理轉換為自動化框架管理。

### 核心概念
*   **角色定位**：CMake 作為建置系統產生器 (Build System Generator)，負責協調檔案連結、相依性管理 (如 JUCE) 與跨平台設定。
*   **自動化優勢**：減少手動設定 VS 專案屬性 (Include Paths, Library Links) 的負擔，讓開發者專注於 DSP 邏輯。

### 兩大操作階段
1.  **Configuration (配置/產生)**:
    *   指令範例：`cmake --preset default`
    *   行為：讀取 `CMakeLists.txt` 與 `CMakePresets.json`，在 `out/` 資料夾中生成具體的建置檔（如 VS Solution 或 Ninja 檔）。
    *   關鍵時機：**新增或重新命名檔案**、修改 `CMakeLists.txt` 或更換建置環境時。
2.  **Build (編譯)**:
    *   指令範例：`cmake --build --preset default`
    *   行為：呼叫編譯器 (MSVC/Clang) 將原始碼編譯成執行檔或插件。
    *   關鍵時機：**修改現有檔案的內容**後進行更新。

## 2. JUCE 插件架構 (JUCE Plugin Architecture)
確立了接下來開發 JUCE 插件時的重點關注對象：

### 核心元件
*   **PluginProcessor (算法核心)**:
    *   負責音訊緩衝處理 (Audio Buffer Processing)、算法邏輯、參數管理。
    *   這是專案的「大腦」，執行主要的 DSP 運算。
*   **PluginEditor (介面設計)**:
    *   負責使用者介面 (UI)、Slider、Button、視覺反饋。
    *   這是專案的「外殼」，處理互動邏輯。
